# Graduate Matrix — Supabase Schema Design

## 1. Scope and principles

This document defines the high-level PostgreSQL schema for Graduate Matrix. It does not define migrations, repositories, authentication, Row Level Security (RLS), React state, save orchestration, or legacy import implementation.

Supabase PostgreSQL will be the system of record for durable operational data. React state will later hold a working copy, unsaved edits, and transient UI state. Active tabs, searches, filters, selected competencies, expanded panels, and modal state do not belong in the database.

The design follows these rules:

- use relational records rather than a whole-application JSON document;
- preserve the approved TypeScript domain boundaries;
- keep evidence, links, cycles, actions, reviews, and audit history normalized;
- treat progression and verification history as durable audit data;
- preserve `candidate_competencies.active_cycle_id → competency_cycles.level` as the only authoritative current level;
- retain historical cycles, including multiple attempts at the same level;
- keep calculated summaries out of storage;
- introduce a small organisation boundary now without building enterprise features;
- keep legacy storage and import concerns outside the operational schema.

The proposed inventory contains **30 tables**: **28 MVP/reference and operational tables**, plus **2 deferred legacy-import tables**.

## 2. Architectural decisions

### Relational operational model

Candidate data is split by domain rather than stored as one JSONB state object. Link entities represent many-to-many relationships. Audit events are append-only. Static definitions remain application-owned unless relational foreign keys materially benefit from a database mirror.

### Identifier strategy

- Operational primary keys use UUIDs generated by the database, normally `gen_random_uuid()`.
- Static framework rows use stable text keys already owned by the application, such as `mcibse-a1`.
- `auth.users.id` remains the authentication identifier and is referenced from an application `user_profiles` row.
- Human-readable references such as `A1` are not primary keys.
- Optional external references use a scoped unique constraint such as `(organization_id, external_reference)`.
- Legacy IDs are not reused as operational primary keys. Phase 12 mappings belong in deferred import tables.

### Timestamp strategy

- Real instants use `timestamptz`, stored in UTC: `created_at`, `updated_at`, `occurred_at`, `assessed_at`, `completed_at`, and sign-off timestamps.
- Calendar-only values use `date`: scheme start, expected application, evidence date, due date, review date, meeting date, and CPD date.
- `created_at` is included where provenance or stable ordering matters.
- `updated_at` is included on mutable rows used by save orchestration and concurrency checks. Append-only audit rows do not need `updated_at`.

### Union values

Canonical union values should initially use `text` columns with named `CHECK` constraints rather than PostgreSQL enums. Adding a value to an enum requires coordinated database migrations and can complicate transactional deployment. Text checks remain explicit, searchable, and easier to evolve. Repository mappers still expose TypeScript unions.

Phase 5 must preserve canonical literals exactly. In particular:

- `CpdCategory`: `T`, `M`, `P`, `E`, `uncategorized`;
- `DevelopmentActionStatus`: `open`, `submitted`, `returned-for-revision`, `completed`, `closed`;
- `CycleStatus`: `locked`, `open`, `paused`, `completed`, `archived`;
- progression approval authority: `mentor`, `mentor-and-manager`.

Hyphenated values must not be rewritten with underscores or renamed in SQL.

### JSONB policy

Only `evidence_entries.structured_sections` is proposed as JSONB. Its expected shape is:

```json
{
  "method": "carr",
  "values": {
    "context": "...",
    "action": "..."
  }
}
```

The keys vary by evidence method and are edited/read as one value; relational columns would create sparse method-specific tables or an unhelpful key/value table. Validate the top-level shape in application mapping and, if feasible, with a lightweight database check. No initial GIN index is recommended because queries are expected to retrieve the document with its evidence row, not search arbitrary section text. Other flexible arrays and relationships are relational.

## 3. Static code-owned data vs database data

The TypeScript source remains authoritative for framework releases and business rules.

| Dataset | Recommendation | Reason |
| --- | --- | --- |
| Competency definitions | Hybrid: code-owned and deterministically seeded into `competency_definitions` | Candidate progress and links need reliable foreign keys and reportable references. |
| Baseline task definitions | Hybrid: code-owned and seeded into `baseline_task_definitions` | Candidate task state needs a stable definition FK and ordering. |
| Standard development tasks | Hybrid: code-owned and seeded into `standard_task_definitions` | Actions need a stable optional template origin. |
| Progression rubric | Code-owned | It drives presentation/business interpretation and is not candidate operational data. |
| Assessment-method definitions | Code-owned | Method codes are small controlled application values. |
| Pathway options and derivation rules | Code-owned | Applicability, derivation, and editability are business rules, not mutable tenant content. |
| Competency target logic and readiness rules | Code-owned | These are pure calculations and must not be executable database data. |

Seeded static tables should carry a `framework_version` or `definition_version` text value and an `is_active` flag. Seed changes must be controlled; administrators should not edit these rows through normal product workflows.

## 4. Tenant and identity boundary

Create `organizations` now. Every candidate belongs to exactly one organisation. This avoids a destructive tenant retrofit while keeping the model small.

`user_profiles` maps one-to-one to `auth.users`, while `organization_memberships` allows a user to participate in more than one organisation. A candidate may optionally link to a user account, so records can be created before an invitation or can remain non-login records.

Mentor, manager, and reviewer assignments use `candidate_relationships`. Each relationship has a role, display name, and optional `user_profile_id`. This supports both authenticated people and named external/uninvited people without copying names into every assessment.

Three role concepts must remain distinct:

1. `WorkflowRole` (`graduate`, `mentor`) is only application workflow/UI mode. It must never be persisted or used as the durable authorization source.
2. `candidate_relationships.relationship_type` (`mentor`, `manager`, `reviewer`) describes a person's relationship to one candidate. It does not automatically grant organisation-wide permission.
3. `organization_memberships.membership_role` is the future organisation-level authorization input. The recommended initial values are `member`, `mentor`, `manager`, and `organization-admin`.

The membership roles are an initial architectural recommendation, not an implemented RLS policy. Phase 5 RLS must reconcile organisation membership with candidate-specific relationships.

Only `organizations`, `organization_memberships`, and `candidates` require direct `organization_id`. Operational child tables derive tenancy through `candidate_id` or their parent chain. This avoids redundant tenant columns. Later RLS and repository queries must join through the candidate boundary. If measured RLS performance becomes problematic, carefully constrained tenant columns can be added later.

No billing, subscription, white-label, or organisation-custom framework tables are included.

30. `legacy_id_mappings` — scoped legacy-to-canonical ID mappings.

## 6. Detailed table design

The column lists below are architectural, not migration-ready SQL.

### Tenant and identity tables

#### `organizations`

- **PK:** `id uuid`.
- **Columns:** `name`, optional `slug`, `created_at`, `updated_at`, optional `archived_at`.
- **Constraints:** unique `slug` when present.
- **Deletion:** application-controlled archive; restrict deletion while candidates or memberships exist.

#### `user_profiles`

- **PK/FK:** `user_id uuid → auth.users.id`.
- **Columns:** `display_name`, `created_at`, `updated_at`.
- **Deletion:** normally `ON DELETE CASCADE` from `auth.users` because this row contains application profile metadata, but audit rows retain actor display snapshots and must not cascade through this table.

#### `organization_memberships`

- **PK:** `id uuid`.
- **FKs:** `organization_id → organizations`, `user_id → user_profiles`.
- **Columns:** `membership_role text`, `created_at`, optional `archived_at`.
- **Checks:** initial values `member`, `mentor`, `manager`, `organization-admin`.
- **Constraints:** unique `(organization_id, user_id)` for a current membership; role check.
- **Deletion:** archive membership; restrict organisation deletion.

#### `candidates`

- **PK:** `id uuid`.
- **FKs:** `organization_id → organizations`; nullable unique `user_id → user_profiles`.
- **Columns:** first name, surname, job title, discipline, employer/team, office location, `scheme_start_date date`, `expected_application_date date`, optional `external_reference`, `created_at`, `updated_at`, optional `archived_at`.
- **Constraints:** unique `(organization_id, external_reference)` when present; one candidate per linked user where `user_id` is present.
- **Deletion:** archive by default. Hard delete only before meaningful child data exists.

#### `candidate_relationships`

- **PK:** `id uuid`.
- **FKs:** `candidate_id → candidates`; nullable `user_id → user_profiles`.
- **Columns:** `relationship_type` (`mentor`, `manager`, `reviewer`), `display_name`, `starts_at`, nullable `ends_at`, `created_at`, `updated_at`.
- **Constraints:** a partial unique constraint for one current relationship of each type per candidate, unless future requirements prove multiple concurrent mentors are needed.
- **Deletion:** historical assignments are ended, not deleted. `user_id` may use `SET NULL` so the named relationship survives account removal.

The repository/domain mapper reconstructs the existing flat `CandidateInfo.mentorName`, `lineManagerName`, and `reviewerName` values from the current relationship rows. Do not add duplicate authoritative name columns to `candidates` merely to match the transitional TypeScript shape.

### Seeded definition tables

#### `competency_definitions`

- **PK:** `id text`, for example `mcibse-a1`.
- **Columns:** unique `reference`, area, objective, behaviours, level expectation, evidence examples, assessment methods, frequency, relevance, notes, framework version, source order, `is_active`.
- **Deletion:** restrict while referenced; deactivate superseded definitions.

#### `baseline_task_definitions`

- **PK:** `id text`.
- **Columns:** title, description, mandatory flag, completion mode, source order, definition version, `is_active`.
- **Checks:** completion mode in `automatic`, `mentor`.
- **Deletion:** restrict while candidate state exists.

#### `standard_task_definitions`

- **PK:** `id text`.
- **Columns:** source wording/template fields, source order, definition version, `is_active`.
- **Deletion:** restrict while actions reference the template.

### Candidate setup and pathway tables

#### `candidate_pathways`

- **PK:** `candidate_id uuid`, also FK to `candidates`.
- **Columns:** professional body, primary outcome, CIBSE membership target, IET membership target, engineering registration target, current membership status, academic route, notes, `created_at`, `updated_at`.
- **Constraints:** exactly one pathway row per candidate. Checks must use the corrected canonical values:
  - professional body: `cibse`, `iet`, `imeche`, `cibse-certification`, `internal`, `other`;
  - CIBSE membership target: `none`, `graduate`, `affiliate`, `lcibse`, `acibse`, `mcibse`, `fcibse`;
  - IET membership target: `none`, `student`, `tmiet`, `miet`, `fiet`;
  - engineering registration target: `none`, `engtech`, `ieng`, `ceng`, `international-later`;
  - primary outcome uses the exact canonical `PrimaryOutcome` literals.
- **Deletion:** cascade only when the candidate itself is legitimately hard-deleted.

Pathway applicability, derivation, normalization, and field editability remain application business logic. Do not persist duplicated visibility, applicability, derived-field, or editability flags.

#### `candidate_pathway_lcc_strands`

- **PK:** `(candidate_id, strand_code)`.
- **FK:** `candidate_id → candidate_pathways.candidate_id`.
- **Checks:** code is one of the code-owned LCC values.
- **Deletion:** cascade with pathway; normal edits insert/delete selections transactionally.

#### `candidate_pathway_specialist_routes`

- **PK:** `(candidate_id, route_code)`.
- **FK:** `candidate_id → candidate_pathways.candidate_id`.
- **Checks:** code is one of the code-owned route values.
- **Deletion:** cascade with pathway.

#### `candidate_baseline_setups`

- **PK:** `candidate_id uuid`, also FK to `candidates`.
- **Columns:** status, `formal_training_started_at`, nullable `formal_training_started_by_user_id`, actor display snapshot, `created_at`, `updated_at`.
- **Checks:** status in `not-started`, `in-progress`, `completed`.
- **Deletion:** cascade only with legitimate candidate hard deletion.

#### `candidate_baseline_tasks`

- **PK:** `id uuid`.
- **FKs:** `candidate_id → candidate_baseline_setups`; `definition_id → baseline_task_definitions`; nullable completed-by user.
- **Columns:** status, note, `completed_at`, completed-by display snapshot, `updated_at`.
- **Constraints:** unique `(candidate_id, definition_id)`; status check.
- **Behaviour:** automatic completion is calculated in application logic. The database stores the resulting task state, never executable rules.

### Competency progression tables

#### `candidate_competencies`

- **PK:** `id uuid`.
- **FKs:** `candidate_id → candidates`; `competency_definition_id → competency_definitions`; nullable `active_cycle_id` using the composite integrity strategy in section 8.
- **Columns:** nullable target-level override, `created_at`, `updated_at`, optional integer `version`.
- **Constraints:** unique `(candidate_id, competency_definition_id)`; target level check.
- **Initialization:** create one row per active competency definition when a candidate enters the framework. This can be transactional and idempotent.
- **Deletion:** restrict once cycles, evidence links, assessments, or events exist; archive the candidate instead.

#### `competency_cycles`

- **PK:** `id uuid`.
- **FK:** `candidate_competency_id → candidate_competencies`.
- **Columns:** level, status, `opened_at`, opened-by user/display snapshot, `completed_at`, completed-by user/display snapshot, completion reason, `created_at`, `updated_at`, optional `version`.
- **Checks:** level in L1–L5; status in `locked`, `open`, `paused`, `completed`, `archived`; completed metadata consistency where practical.
- **Constraints:** unique `(candidate_competency_id, id)` supports the active-cycle composite FK. Do not make `(candidate_competency_id, level)` unique; repeated attempts are valid.
- **Deletion:** restrict once referenced. Historical cycles are retained.

`archived` remains a reserved canonical value. The approved state machine exposes no active archive workflow. Phase 5 may permit the literal in constraints for canonical compatibility, but must not invent an archive RPC, transition, repository command, or UI.

#### `mentor_assessments`

- **PK:** `id uuid`.
- **FKs:** candidate, candidate competency, and cycle; nullable assessor user.
- **Columns:** assessment status, progression recommendation, next action, `assessed_at`, assessor display snapshot, `created_at`, `updated_at`.
- **Constraints:** all three parent relationships must agree; enforce through composite keys or transaction/repository validation. Status and recommendation checks.
- **Deletion:** audit-sensitive; no routine hard delete. Corrections create a new assessment or use controlled administrative correction metadata later.

Completion authority does not belong on `mentor_assessments`. The assessment establishes demonstrated/recommendation state. Mentor completion, manager sign-off, confirmation, decision reason, and evidence basis belong on the completion `progression_events` row so the transition fact exists once.

#### `competency_cycle_reviews`

- **PK:** `id uuid`.
- **FKs:** candidate, candidate competency, cycle, nullable reviewer user.
- **Columns:** status, recommendation, next action, `reviewed_at`, reviewer display snapshot, `created_at`.
- **Deletion:** append-only in ordinary operation; restrict parent deletion.

#### `progression_events`

- **PK:** `id uuid`.
- **FKs:** `candidate_id`, `candidate_competency_id`, `cycle_id`; nullable `destination_cycle_id`; nullable performed-by, mentor, and manager users.
- **Common columns:** `event_type`, `occurred_at`, performed-by display snapshot, reason, imported flag/source reference.
- **Typed columns:** level, from level, to level, previous cycle ID, destination cycle ID, mentor approved timestamp/display name, manager sign-off timestamp/display name, `manager_signoff_confirmed`, approval authority, and `evidence_basis`.
- **Checks:** event type in the canonical event union; event-specific checks require and prohibit relevant typed fields. For example, completion requires from level, mentor approval, manager sign-off, confirmation, and `mentor-and-manager` authority; opening requires level; reset requires a source cycle.
- **Model choice:** one relational event table with typed nullable columns. There are few stable variants, common chronological queries dominate, and subtype tables would add joins without removing many nullable fields. JSONB is not needed.
- **Mutation:** append-only. No `updated_at`; administrative corrections should append a correcting event or use a separately audited process.
- **Deletion:** restrict parent deletion and prohibit ordinary event deletion.

`managerSignoffConfirmed` and `evidenceBasis` are now canonical durable fields on `CycleCompletedEvent`; their database columns are required for `cycle-completed` events. Mentor assessment status and recommendation remain in `mentor_assessments`, while transition approval and the evidence basis for completion remain in the progression event. Do not duplicate these completion facts across both tables.

`mentor-and-manager` is the only currently active source-backed completion authority. `mentor` remains a reserved canonical literal; Phase 5 must not create a mentor-only completion workflow unless later approved business logic activates it. Likewise, `cycle-archived` remains reserved in the canonical event union but does not justify an archive workflow in Phase 5.

### Evidence and action tables

#### `evidence_entries`

- **PK:** `id uuid`.
- **FK:** `candidate_id → candidates`.
- **Columns:** evidence date, claimed level, project reference/type, RIBA stage, title, description, outcome, method, `structured_sections jsonb`, systems text array or normalized later if systems become managed reference data, evidence-derived CPD hours/category/sign-off fields, `created_at`, `updated_at`.
- **Checks:** claimed level, evidence method, CPD category, finite CPD hours where supported.
- **Verification status:** do not store a second authoritative status. Derive current status from the latest `evidence_verification_events` row; no history means `unverified`.
- **Domain compatibility:** `EvidenceEntry.verificationStatus` remains temporarily in the TypeScript shape. Database mappers populate it from the latest event, or `unverified` when no event exists. It is not a persisted fallback column.
- **Deletion:** prefer archive metadata only if product deletion becomes necessary. Once verification or progression relies on evidence, restrict hard deletion.

#### `evidence_competency_links`

- **PK:** `id uuid`.
- **FKs:** evidence and competency definition; nullable accepted-by user.
- **Columns:** link type, `accepted_at`, accepted-by display snapshot, `created_at`.
- **Constraints:** unique `(evidence_id, competency_definition_id)`; link type in `primary`, `accepted`, `suggested`; partial unique index for at most one primary link per evidence.
- **Deletion:** link removal is application-controlled. Audit requirements may later justify link-history events, but they are not invented here.

#### `evidence_verification_events`

- **PK:** `id uuid`.
- **FKs:** `evidence_id → evidence_entries`; nullable actor user.
- **Columns:** event type, actor display snapshot, `occurred_at`, nullable reason.
- **Checks:** event type in `verified`, `verification-revoked`, `reverification-required`.
- **Mutation/deletion:** append-only; restrict evidence deletion.
- **Current state:** latest event by `(occurred_at, created_at, id)` determines current verification state. Repository transactions should prevent ambiguous simultaneous updates.

For Phase 12 imports, legacy evidence with a non-default status but no reconstructable history must receive a synthesized backdated event. A legacy verified value creates a `verified` event; a legacy reverification-required value creates a `reverification-required` event. The importer should retain import provenance, use the best available historical timestamp and actor, and record an explicit import fallback when exact details are unavailable. It must not introduce a permanent current-status fallback column.

#### `development_actions`

- **PK:** `id uuid`.
- **FKs:** candidate, candidate competency, cycle, nullable standard task definition, nullable carried-forward source action, nullable actor users.
- **Columns:** title, notes, owner, priority, status, `due_date date`, submission/completion timestamps and actor snapshots, archive timestamp/actor/reason, `created_at`, `updated_at`.
- **Checks:** owner, priority, status, and archive metadata consistency. The consistency rules are a new database/repository integrity improvement, not an invariant already enforced by Phase 3 logic:
  - `completed_at` must be non-null when status is `completed`;
  - `completed_by` should be non-null when `completed_at` is non-null;
  - `archived_at` may coexist with `completed` or `closed`, because archive is orthogonal to completion history;
  - `archived_by` should be non-null when `archived_at` is non-null;
  - `archive_reason` follows the approved archive workflow requirements;
  - archived actions cannot accept ordinary future workflow transitions through repositories/services.
- **Deletion:** use existing archive metadata. Do not create a deleted-actions table. Restrict hard deletion after evidence links or carry-forward descendants exist.

Legacy/imported inconsistencies must be handled by an explicit import policy rather than by weakening ordinary operational writes or rejecting historical data before Phase 12 has a repair strategy.

#### `evidence_action_links`

- **PK:** `id uuid`.
- **FKs:** evidence and development action; nullable created-by user.
- **Columns:** created-by display snapshot, `created_at`.
- **Constraints:** unique `(evidence_id, development_action_id)`.
- **Deletion:** cascade when an uncommitted link parent is legitimately deleted; otherwise application-controlled unlinking.

### Reviews, meetings, and CPD tables

#### `candidate_reviews`

- **PK:** `id uuid`.
- **FKs:** candidate and nullable reviewer user.
- **Columns:** `reviewed_on date`, reviewer display snapshot, outcome, `next_review_date date`, notes, `created_at`, `updated_at`.
- **Checks:** review outcome values.
- **Deletion:** audit-sensitive; no routine hard delete.

#### `meetings`

- **PK:** `id uuid`.
- **FKs:** candidate; nullable `review_id → candidate_reviews`; nullable created-by user.
- **Columns:** meeting date, type, attendees, duration, notes, outcome, candidate comment/reflection, created-by snapshot, `created_at`, `updated_at`.
- **Deletion:** restrict candidate deletion; `review_id` may use `SET NULL` so a meeting survives review correction/removal.

#### `cpd_entries`

- **PK:** `id uuid`.
- **FKs:** candidate; nullable sign-off user.
- **Columns:** CPD date, title, hours, category, description, outcome, sign-off timestamp/display snapshot, `created_at`, `updated_at`.
- **Checks:** category values and finite numeric hours. Negative hours remain permitted because the approved source logic preserves them.
- **Deletion:** restrict after sign-off or competency linking; otherwise use application-controlled correction/archive if later required.

Ordinary standalone CPD persists only canonical categories: `T`, `M`, `P`, `E`, `uncategorized`. Evidence-derived CPD begins with the free-text `EvidenceCpdMetadata.category`, so repository/import mapping must normalize it before persistence or calculation boundaries. Legacy sentinels such as `—` map to `uncategorized` or the source-backed competency fallback. Database checks must not be widened to accept obsolete sentinel values.

#### `cpd_competency_links`

- **PK:** `id uuid`.
- **FKs:** CPD entry and competency definition; nullable accepted-by user.
- **Columns:** link type, `accepted_at`, accepted-by display snapshot, `created_at`.
- **Constraints:** unique `(cpd_entry_id, competency_definition_id)`; type in `accepted`, `suggested`.
- **Deletion:** application-controlled unlinking.

#### `cpd_attachments`

- **PK:** `id uuid`.
- **FK:** `cpd_entry_id → cpd_entries`.
- **Columns:** object/storage key or name, display filename, `added_at`, optional added-by user.
- **Constraints:** unique `(cpd_entry_id, storage_key)` when storage exists.
- **Deletion:** cascade with a legitimately deleted unsaved CPD record; storage deletion must be explicitly orchestrated later.

### Deferred import tables

#### `legacy_import_batches`

Stores organisation, source filename/type, checksum, initiated-by user, timestamps, status, and error summary. It is not an operational snapshot table.

#### `legacy_id_mappings`

Maps `(import_batch_id, entity_type, legacy_id)` to canonical table/UUID. Unique within the batch and entity type. It supports idempotent Phase 12 imports without contaminating operational IDs.

## 8. Authoritative competency-cycle model

`candidate_competencies` is the candidate-specific root. Its `active_cycle_id` is nullable and is the only pointer to the authoritative current level. The level is read from the referenced `competency_cycles.level`. Do not add `current_level`, `mentor_level`, `self_level`, or `active_cycle_level` columns.

The circular relationship is intentional:

1. `competency_cycles.candidate_competency_id` references its owner.
2. Add `UNIQUE (candidate_competency_id, id)` to cycles.
3. Add a composite FK from `(candidate_competencies.id, candidate_competencies.active_cycle_id)` to `(competency_cycles.candidate_competency_id, competency_cycles.id)`.
4. Keep `candidate_competencies.active_cycle_id` nullable.
5. Add the composite ownership FK after both tables exist. It may be `DEFERRABLE` to support tightly controlled multi-row progression transactions, but deferrability does not replace correct operation ordering.

This prevents one candidate competency from pointing to another competency's cycle. PostgreSQL migration ordering must create the tables first and add the circular FK afterward.

Initial cycle creation should use one controlled transaction:

1. insert the candidate competency with `active_cycle_id = NULL`;
2. insert its owned competency cycle;
3. update `active_cycle_id` to that cycle;
4. commit after the composite FK and expected ownership checks pass.

A post-insert update in the same transaction is simpler than relying on deferred validation for every workflow. Use deferrability only where a specific atomic transition genuinely needs temporary statement-level ordering flexibility.

Multiple cycles at the same level are allowed. Active reset and earlier-level reopening create new cycle rows rather than overwriting history. Completed L5 may remain selected through `active_cycle_id` for source fidelity even though its status is completed.

The database should represent source reset semantics as:

- canonical cycle status `locked`;
- a durable `cycle-reset` progression event for that cycle;
- a `cycle-opened` event and replacement row when active reset creates a same-level replacement.

Do not persist Phase 3E's transition-local `resetLockedCycles` metadata. Reset/locked meaning is derived from the event history plus the locked cycle. This avoids adding a parallel cycle-state model. If future querying shows this derivation is too costly, a narrowly constrained `reset_at` cache may be considered later, but it is not part of MVP.

## 9. Audit and progression history

`progression_events` is the only progression-history store. It supports chronological queries by candidate, competency, cycle, and timestamp. Event IDs are unique UUIDs. Actor UUIDs are nullable to support imported or unregistered actors, while display-name snapshots preserve what was recorded at the time.

Completion facts live on the `cycle-completed` event:

- source and destination cycle;
- from/to level;
- mentor approval identity and time;
- manager sign-off identity and time;
- canonical `managerSignoffConfirmed`;
- approval authority;
- decision reason;
- canonical `evidenceBasis`.

These facts should not also be copied onto mentor assessments. Cycle convenience fields such as `completed_at`, `completed_by`, and completion reason are operational cycle state and must be written in the same transaction as the audit event. The event remains the audit source; the cycle fields support current-state queries.

Evidence verification history is similarly append-only and is the durable authority. A revoked or reverification-required event never deletes a prior verified event. New database-backed evidence with no events is `unverified`; there is no durable fallback status column.

Imported legacy events are inserted into the same canonical event table after mapping, with explicit import provenance. Legacy evidence with a non-default verification status but missing history receives a synthesized verification event during Phase 12. There is no legacy progression-history or verification-status fallback table.

| Progress percentages and dashboard KPIs | candidate competencies, cycles, evidence, actions, reviews, CPD |

## 11. Transaction boundaries

### Routine low-risk saves

Later debounced autosave may suit candidate descriptive fields, pathway notes, draft evidence text, draft action text, and unsent meeting/CPD details. Each save should update a narrowly scoped row rather than the entire workspace.

### Explicit controlled saves

Use explicit user actions for mentor assessments, evidence verification, CPD sign-off, formal reviews, baseline completion, action completion/archive, and relationship assignment. These changes have workflow or audit meaning.

### Transactional multi-record operations

The following should use a database transaction, likely through a narrowly typed PostgreSQL function/RPC when repositories alone cannot guarantee atomicity:

- initialize all candidate competency rows;
- complete L1–L4, update the source cycle, create the destination cycle, update `active_cycle_id`, insert the completion event, and apply carry-forward action changes;
- complete L5 and insert its completion event while retaining the L5 pointer;
- active reset: lock/reset the old cycle, create the same-level replacement, update the pointer, and insert reset/open events;
- reopen an earlier level: validate represented cycle IDs, pause applicable cycles, insert pause/reopen events, create the earlier cycle, and update the pointer;
- paused-cycle recovery: update cycle state, insert reset event, and coordinate separately validated evidence/action recovery changes;
- create evidence with competency/action links;
- verify or revoke evidence and append the verification event;
- create a progression completion event with mentor and manager authority details.

Transactions must accept all business inputs explicitly. Database functions should enforce structural integrity and concurrency, while pure TypeScript remains the business-rule source.

## 12. Concurrency strategy

Use `updated_at` for human-readable update/audit timing and ordinary optimistic compare-and-swap where timestamp precision is sufficient. Updates include the previously read timestamp and fail when zero rows match.

Recommended practical split:

- use `updated_at` compare-and-swap for ordinary forms, evidence, actions, pathways, assessments, and candidate details;
- do not add integer versions to every mutable table by default;
- reserve integer `version` for candidate competencies and competency cycles only if transition/concurrency testing demonstrates a concrete benefit;
- use database transactions and row locks (`SELECT ... FOR UPDATE`) for progression transitions;
- keep append-only audit events free of update races;
- surface conflicts to users rather than silently overwriting newer data.

Autosave must not retry a stale write blindly. Transition RPCs should require the expected active cycle ID and, preferably, expected candidate-competency version.

## 13. Foreign-key and deletion strategy

- Organisation deletion: `RESTRICT`; archive instead.
- Membership/profile deletion: archive memberships; profile removal may null optional actor links while snapshots remain.
- Candidate deletion: archive once meaningful data exists; child FKs generally `RESTRICT`.
- Pathway option rows and baseline task rows: `CASCADE` only with a legitimate hard deletion of their direct candidate-owned parent.
- Seeded definitions: `RESTRICT`; deactivate rather than delete.
- Candidate competencies and cycles: `RESTRICT`; preserve history.
- Progression and verification events: `RESTRICT` and append-only.
- Evidence: restrict deletion after links or verification events; unlink deliberately.
- Development actions: use archive metadata; restrict deletion after evidence links or carry-forward descendants.
- Reviews and assessments: restrict routine deletion.
- Meetings: candidate FK restrict; optional review FK `SET NULL`.
- CPD: restrict signed-off/linked records; attachment metadata may cascade only with a legitimate CPD hard delete.

Database cascades should be limited to dependent selections or metadata that have no independent audit meaning. An application-level candidate purge, if ever legally required, must be a separately authorized and audited operation.

## 14. Constraints and indexes

### Important constraints

- unique organisation slug when present;
- unique current organisation membership `(organization_id, user_id)`;
- unique candidate external reference within organisation when present;
- one candidate pathway and baseline setup per candidate;
- unique baseline task state `(candidate_id, definition_id)`;
- unique candidate competency `(candidate_id, competency_definition_id)`;
- composite active-cycle ownership FK;
- no uniqueness constraint on cycle level attempts;
- unique evidence-competency pair and at most one primary link per evidence;
- unique evidence-action pair;
- unique CPD-competency pair;
- unique progression and verification event IDs;
- text checks for canonical statuses, levels, link types, roles, and event types;
- event-specific checks for required typed progression columns;
- completion timestamp/actor consistency checks where they do not prevent legacy import staging.

Cross-table facts such as “assessment candidate matches cycle candidate” may require composite FKs or transaction-level validation. Prefer composite ownership keys for high-risk progression tables.

### Initial indexes

- `candidates (organization_id, archived_at)`;
- `organization_memberships (user_id, archived_at)`;
- `candidate_relationships (candidate_id, relationship_type, ends_at)`;
- `candidate_competencies (candidate_id)`;
- `competency_cycles (candidate_competency_id, created_at desc)` and `(candidate_competency_id, level)`;
- `progression_events (candidate_id, occurred_at desc)` and `(candidate_competency_id, occurred_at desc)`;
- `evidence_entries (candidate_id, evidence_date desc)`;
- `evidence_competency_links (competency_definition_id, link_type)` plus the unique evidence pair index;
- `evidence_verification_events (evidence_id, occurred_at desc)`;
- `development_actions (candidate_id, status, due_date)` and `(candidate_competency_id, archived_at)`;
- `mentor_assessments (cycle_id, assessed_at desc)`;
- `competency_cycle_reviews (cycle_id, reviewed_at desc)`;
- `candidate_reviews (candidate_id, reviewed_on desc)`;
- `meetings (candidate_id, meeting_date desc)`;
- `cpd_entries (candidate_id, cpd_date desc)`;
- `cpd_competency_links (competency_definition_id, link_type)`.

Do not initially index notes, descriptions, every status column, JSONB sections, or low-cardinality fields independently.

## 15. MVP schema vs deferred schema

### Eventual MVP table inventory

The 28 tables numbered 1–28 are the eventual MVP schema because they establish tenant ownership, identity, candidate setup, the complete approved domain, FK integrity, and audit history. Core integrity is not deferred, but these tables must not be delivered as one untested initial migration.

### Phase 5 staged vertical slices

The first slice proves the highest-risk architecture end to end:

```text
organization
→ user profile and organization membership
→ candidate
→ seeded competency definition
→ candidate competency
→ competency cycle
→ composite active-cycle ownership integrity
→ basic create/load/save
→ first transactional cycle operation
```

This slice must test cross-competency pointer rejection, nullable initial pointers, transactional cycle creation, optimistic conflict handling, and repository/domain mapping before expanding the schema. Subsequent slices add the remaining MVP domains without changing the ownership model.

### Deferred

- `legacy_import_batches` and `legacy_id_mappings` until Phase 12;
- snapshot records;
- manual export history;
- browser/file-system compatibility;
- advanced attachment/storage lifecycle records beyond CPD attachment metadata;
- organisation-specific framework customisation;
- notification delivery history;
- analytics/materialized views;
- legal purge workflow metadata.

No legacy ID columns are required across every operational table. Import mappings provide compatibility later.

## 16. Open questions and recommended decisions

1. **Candidate relationships:** Can multiple mentors be active concurrently? Recommend one current mentor, manager, and reviewer initially, with history retained through `starts_at`/`ends_at`. Relax the partial unique constraint later if the product requires teams.
2. **Completed L5 pointer:** The canonical/source behaviour keeps completed L5 selected. Preserve it. Document that `active_cycle_id` means the authoritative selected/current cycle, not necessarily status `open`.
3. **Reset state:** Derive reset/locked from the locked cycle and `cycle-reset` event. Do not add `reset` to persisted status or store transition-local metadata.
4. **Completion cycle fields versus event facts:** Keep cycle completion fields as transactionally maintained current-state columns and the progression event as immutable audit authority. Never allow them to be written independently.
5. **Actor deletion:** Retain actor display snapshots and permit nullable user FKs so historical audit remains intelligible if accounts are removed.
6. **Evidence current verification status:** Prefer derivation from verification events. If performance later requires a cached status, update it only inside the same transaction and treat events as authoritative.
7. **Evidence systems:** Keep the canonical text array initially because systems are descriptive values without an approved managed vocabulary. Normalize only if filtering/governance requirements emerge.
8. **CPD target:** The canonical state does not define a durable candidate target record. Keep targets code/config-owned until requirements establish whether they vary by candidate, organisation, or reporting year.
9. **Framework versioning:** Seed stable definition IDs and retain inactive definitions. If future framework releases change meaning materially, add a framework/version parent rather than mutating historical definitions in place.
10. **RLS performance:** Begin with tenancy derived through candidate FKs. Add redundant `organization_id` only after measuring query/RLS cost and pairing it with consistency constraints.
11. **Organisation membership authorization:** Begin with `member`, `mentor`, `manager`, and `organization-admin`, then validate the permission matrix during Phase 5. Candidate relationship roles remain separate and may narrow access without automatically granting organisation-wide rights.
12. **Transitional domain compatibility:** Adding required completion fields to `CycleCompletedEvent` means every constructor/mapper must supply them. Phase 5 repository work must preserve this canonical contract rather than introduce nullable database fallbacks.

## 17. Phase 5 implementation sequence

Recommended dependency-aware vertical slices:

1. Establish Supabase project/client boundaries without application workflows.
2. **Foundation slice:** implement organisations, user profiles, memberships, candidates, and the minimum typed create/load/save repository path.
3. **Cycle-integrity slice:** seed competency definitions; add candidate competencies and cycles; add the composite FK through a post-table migration; prove `NULL → owned cycle` pointer update in one transaction; reject cross-competency pointers.
4. Add the first transactional cycle operation with expected active-cycle checks, row locking, audit event insertion, and integration tests before adding more tables.
5. Add candidate relationships and mapper reconstruction of flat `CandidateInfo` relationship names.
6. Add pathways and normalized pathway selections using exact canonical checks.
7. Add baseline definitions, setup, and candidate task state.
8. Add the remaining progression events and reset/reopen/complete transaction boundaries. Do not add archive or mentor-only completion workflows.
9. Add evidence, competency links, and authoritative verification events, including mapper-derived domain verification status.
10. Add development actions and evidence-action links with the new repository/database consistency rules.
11. Add mentor assessments and competency-cycle reviews.
12. Add whole-candidate reviews and meetings.
13. Add CPD entries, normalized category handling, competency links, and attachments.
14. Expand typed mappers/repositories and `updated_at` compare-and-swap domain by domain.
15. Add integer versions only to candidate competencies/cycles if transition tests justify them.
16. Implement RLS using organisation membership plus candidate relationships, then test cross-tenant isolation.
17. Validate audit immutability, transition atomicity, canonical literal round-trips, and repository/domain mapping before React save orchestration begins.

RLS comes after schema relationships are stable, but before production data access is enabled. Legacy import tables and adapters remain deferred to Phase 12.
